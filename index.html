IOT Environmental Sensors

A rough overview of the final project architecture.

Project Summary

This is a project that I’ve been working on in my spare time with another electrical engineering student who is working predominantly on the sensors themselves. I’ve been responsible for the system architecture, as well as setting up the website, server and writing the APIs. Over the course of the project a number of different sensors have been designed that record a wide range of different signals such as temperature, CO2 levels, wind speed and many others. This required that the system be flexible enough to manage a range of sensors with different requirements.

Project Breakdown (My Parts)

Sensor to server communication:

For this we decided on using the MQTT protocol as it is a publish-subscribe protocol rather than a client-server protocol meaning that we would have the ability to send new data to the frontend in real time without having to constantly poll the server to see if new data has been received for a particular sensor. I also designed the requests that sensors could make, as well as implementing the server side functionality for dealing with these requests.

Databases:

Since this project required the storage of a large quantity of time-series data I opted to use MongoDB for storing the data produced by the sensors due to it’s flexibility allowing for the storage structures to be very easily changed in the future. It’s flexibility was also ideal for storing the time-series data which would look different depending on the sensor producing it.
In order to store information about user accounts and sensors metadata I opted to use MariaDB as it’s ideal for relational data and optimised for relational queries.

Data API:

This REST API is written in PHP and was designed to make it easy for the frontend to fetch a user’s data. Building separately from the website dashboard also means that in the future if we decide to add an app or allow other webservices access to the sensor and user data it will be a lot easier to do so.

Frontend:

To enable users to view and analyse the data their sensors had collected, I built a website dashboard shown below. It integrates with the API mentioned above and also allows the user to easily manage their sensors, locations, and profile information.

The main website dashboard for analysing the data.

Algorithmic Trading Experimentation

Project Summary

This is a project that I've recently started as it's a good project for developing my skills in finance and algorithm design. It also provides my with an oppurtunity to further develop my machine learning skills by working with real world data. I started this project by exploring trading signals. To start with I look into the simple moving average and the exponential moving average. By using different window sizes and monitoring the cross over points, one can gain an insight as to whether a market is stable or breaking out.

After trying this I looked into support / resistance indicators. The theory behind this is that the support line indicates the lowest price that the stock has been at in the last window size (eg. 200 days). The resistance line marks the highest price that the stock has been at in the the last window size. The idea is that the future stock price is unlikely to breach these lines and thus if it spend consecutive days in the support tolerance then the stock is about to rise and the algorithm should buy. If the price stays within the resistance tolerance band for consecutive days then the algorithm should sell as the stock price is about to decrease. This is a very simple algorithm but can be successful with a select few stocks. An image of it being applyed to historical Google stock data is shown below.

The graph shows how support and resistance indicators could be used to help a trading system decide when to buy and sell stocks.

I also tried applying a linear regression model and k-nearest neighbours model to Google's historical stock data.

Supermarket Sales Forecasting

Introduction

This project was done after coming across a dataset used by one of the introductory Kaggle competitions (Link to Kaggle). The dataset contains sales information about the sales of fifty four convenience stores across Ecuador between 2013 and 2017. The dataset contains quite a lot of additional information about the stores, as well as some of the major events happening in Ecuador during the period that the data was recorded. For this reason I thought it would be fun analyze this particular dataset and see what data analysis techniques I could learn along the way.

Preliminary Exploration of the Main Time Series Data

The main dataset contains the following columns:

id, date, store_nbr, family, sales, onpromotion

A complete description of the data can be found at the link above, but most is self-explanatory. The family column is a product category, the sales is the total number of products in that particular product category sold at the specified store and date. The onpromotion column is the number of those sales that were on promotion at the time.

A good starting point is to get a basic summary of the dataset. This is shown below:

Training Data Summary.

From this alone we can see that we have 54 stores as well as sales that range from 0 to 124,717 however the 50th percentile only 11, and the 75th percentile is only 195 suggesting we have mostly smaller stores with a couple of big ones. If we plot (the right hand graph below) all of the total daily sales against the store that generated them, we can see that this claim is indeed true. The page listed in the introductory section also provides a dataset that contains the location of each store so it would be interesting to include this in the analysis later on to see how location influences the size of the daily sales figures.

The left hand graph shown below is a scatter plot of the daily sales sorted by product category. From this we can see that the majority of categories have very low numbers of sales, with only a few major categories that dwarf the others. This makes sense when you see that the main categories are food and drink related as one would suspect. I will explore later on how these category sales change over time and during major event, as well as how the location of the store may impact the ratio in which goods are sold.

Plots showing all of the sales in the dataset arranged by product category and store ID.

Sales Forecasting (GBM Model)

The official Kaggle competition requires one to forecast 16 days of future sales, using Root Mean Square Log Error (RMSLE) as the error function. For this reason I decided to do the same and submit my predictions to Kaggle for calculating the error.

The first model that I tried to use was from the lightgbm python library which simplifies the implementation of a gradient boosted trees. However before plugging the training the model, I had to develop some features that I thought would be useful. Besides the ones that are already part of the train dataset, I exploded the date to create a day of month, month of year, and year column. I also added day of year and is whether the current day is a weekend to see if this had any influence on the accuracy of the model. I then added a 7 day, 14 day, and 28 day lag. I finally also had to encode the product family as currently it’s categorical. This just involved assigning values to each product category.

This first model had a validation RMSLE of 0.914. However when I submitted it on Kaggle for the test data it had a RMSLE of 2.182. This could either be because of an error in my model, or because the test data contained a week of sales that was very unusual or different from the validation weeks. I will have to look into this at a later point but for now I decided to look at the contribution that each feature was having towards the final model by looking at the feature weights. The following is a graph of these weights:

Feature gains for the first forecasting attempt.

In order to improve this score the first thing I decided to do was run the model on the log of the sales instead of the absolute sales. The reason being that different product families have quite large difference in scale for obvious reasons, and by taking the log this should make the range of values for each product type more similar.

After implementing this change, still using the same features, the validation RMSLE more than halved to 0.423. The submission RMSLE also dropped down to 1.376 which is a massive improvement. The feature gains stayed fairly similar.
